
# JavaScript Style Guide

This is the style guide for coding and editing JavaScript here — the user telling you how to write it. It governs your choices about coding, commenting, documenting, and refactoring. Your default modern style is already good; this document is the delta: where we deliberately differ from prevailing practice, and why. Where it's silent, good current practice applies.

By JavaScript we mean code and comments as one integrated system. The runtime reads only the code; every other reader reads the two as a single text. That's always been true for humans, and now it's true for agents: drop a capable agent into a repo with the comments stripped out, and its results suffer, because the comments were carrying half the meaning. The aim everywhere: code that runs reliably, is factored efficiently, and reads lyrically, for humans and agents alike.

Two ideas underneath everything here. First, simplicity: Python states it as a core value and JavaScript should — simple is better than complex. Reach for the plain pattern over the fancy one, and when code grows complicated, break it into smaller, plainer parts rather than adding cleverness. Second, attention: a reader's attention is what every choice here spends or serves. That's not a rule about writing less — comments enjoy a huge budget in this style; by weight they rival the code. It's a rule about balance: say the right amount in the right places, so understanding arrives as quickly as possible — and keep a few marks rare, so that when one appears, it means something.

This document positions itself on the map of styles you already know — your own house style, bootcamp-and-bank JavaScript, the linter defaults and Prettier, the clean-code and good-parts schools. Where we agree with a school, we say so and take the ally; where we oppose one, the opposition is deliberate, and argued on the spot with real reasons. Knowing what the mainstream does, and choosing with intent, is part of the style.

## Layout

- Indent with tabs; editors show them two wide. The JavaScript mainstream settled on two spaces; we side with Go, whose gofmt chose tabs: a tab is one keystroke, one character, and every reader sets their own width.
- Never wrap lines to a column width; assume a graphical editor with soft wrap on. Artificial wrapping — Prettier's default posture — turns a one-word change into a multi-line diff, and wastes attention counting columns and choosing break points.
- Do align neighboring lines vertically: when adjacent lines share structure, pad with spaces so like parts sit in the same columns. The eye diffs aligned lines instantly — seeing the differences is understanding them. These two rules together are why no formatter runs here: a tool that owns the whitespace erases information we deliberately write into it.
- Single quotes for strings; backticks when interpolating; double quotes inside comments and prose.
- Omit semicolons except between two statements on one line — which we do when they're one thought: `x = 0; y = 0//reset the cursor to the origin`. The semicolon-free camp, StandardJS, has this right: automatic insertion is specified behavior, proven at scale, and the code is quieter without them.
- A short if fits on one line without braces: `if (!row) return false` — or with braces holding two short statements: `if (found) { count++; last = x }`. Strict shops lint braces onto every if, citing Apple's goto-fail bug — but that bug lived in an unbraced body on its own line, a form we never write. A guard and its action on one line is a single readable sentence, with no room beneath it for a second statement to sneak in.
- Prefer if statements over ternaries. A ternary survives only flat and simple, picking between two values; never nested, never carrying logic.
- Trailing commas in multi-line literals, so reordering or adding a line never touches its neighbor.
- Blocks get spaces inside braces `{ like this }`; object literals don't `{key: 'value'}`.
- Comments are conversational lowercase for the same reason syntax highlighting paints them dim gray: to let the code, and its capitalization, stand out. The one exception is the essay, described with the other comment forms below, which moves into a `/* */` block and normal English prose.

## Names

- Name things with complete, correctly spelled words. Abbreviations are banned outright — local, never dev; configuration, never cfg. Treat an abbreviation that's a proper name out in the world, like HTML or URL, as a word: someUrl, someHtml, someId.
- Functions are verbNoun, like validateEmail and parseHeader; getters can be bare nouns.
- A verb prefix is a promise: if is* functions answer yes-or-no, every is* function answers yes-or-no. A reader who has learned one member of a family has learned them all, so no function ever breaks its prefix's contract.
- Domain families lead with the shared noun so they sort and read together: credentialOauthSet, credentialOauthGet, credentialOauthRemove.
- Mark a private helper — a function serving a nearby exported one — with a leading underscore.
- Single-character names are correct in two places. First, conventional carriers a codebase establishes for itself — picture a project where every validation function returns a result object the code always calls v; after you've met v twice, it never needs introducing again. That letter is a convention the project made, not one to import. Second, short-lived locals whose whole life fits on a screen, introduced by a comment; the comment is the real description, and a long name would be a worse, endlessly repeated version of it. The descriptive-names school tolerates single letters only in tiny scopes, and its remedy everywhere else is a longer name, intention revealed by identifier alone. Ours is the opposite: name it briefly, describe it richly, once, in the comment where it's born. A name is repeated at every use; a description shouldn't be.
- Case follows the environment: kebab-case in css, snake_case in sql (our_tables have column_names), camelCase in js. ALL_CAPS is banned — it reads as a shout and edits with clumsy caps lock. Where the mainstream guides prescribe SCREAMING_SNAKE for a constant, write const camelCase at the top of the file instead. Don't reach for capitals without asking.

## Comments

The signature of this style is comment density: get ready to comment far more than you normally would, and differently. This is a considered rejection of an orthodoxy you've absorbed. Clean Code taught that a comment is a failure to express yourself in code, and a generation of review culture inherited the reflex — comments as apologies, deleted on sight, replaced by ever-longer names and ever-smaller functions. We hold with the older and better school — Knuth called it literate programming, a program written for people and only incidentally for the machine: prose can carry intent, context, and caution that code cannot, and refusing that channel throws away half the bandwidth of the file. The bar to write toward: comments so complete that given only a function's name, parameters, and comments — body stripped away — a capable engineer or model could write a functional equivalent.

Form: the comment sits at the end of the line it's about, no space around the slashes:

	let a = []//the array we'll fill with results, sort, and return

Lowercase, no closing period. Two clauses get a semicolon:

	//the address is gmail; apply the extra normalization rules

Several sentences form a short paragraph — capitals and periods between sentences, but the same lowercase open and bare end:

	//the first note. And a second, a clarification. Perhaps a third situation to describe, as well

Comments come in four sizes, small to large: on the end of a line; introducing a group of lines; above a function; and the essay.

The first two live inside function bodies. Line comments go to the right of code, not above, so comments stay plentiful without lengthening the file. Blank lines split a longer function into conceptual groups; the reader sees two groups and knows the function works in two parts, then reads to learn what each part is. A single-line comment above a group is right when it speaks at the group's altitude — what these lines accomplish together. A comment about one line goes on that line.

Above a function, the same single-line style one step zoomed out: what this is for, plus anything clever, unique, or tricky. Javadoc-style blocks are the anti-pattern — formulaic and pedantic, covering the trivial and the tricky at equal weight until the important part is lost in the average, sitting atop code the reader still has to figure out. Comment instead like the engineer who knows this area talking to a capable newcomer: here's what you actually need to know.

Fourth, the essay. When a larger system needs explaining at a larger level — an overall design, a novel mechanism running through the whole codebase, choices and their reasons — the comment grows into a `/* */` block and switches to normal English prose, capitals and all. An essay is written like a little document but is not one: it lives with the code, at the best place it can sit, and it runs a half screenful, not pages. It speaks from ten thousand feet, and it's the one place comments are allowed to remember — the history, how we got here, the alternatives weighed and set aside. Essays span larger time and space; every smaller comment stays tight to now.

What to say: never restate the line. Positioned at the line, a comment gets where for free, so it spends all its words on what that line is doing in service of the surrounding task. Writing good comments means balancing three tensions, deliberately:

**Comment, or let code stand alone.** The test is whether the comment speeds understanding: would a reader move through this stretch faster, easier, and more accurately with it than without it? Trivial code commented is clutter — it slows the eye and dilutes the comments that matter. But an unfamiliar pattern uncommented is worse: every future reader stops to reverse-engineer what one sentence would have handed them, and a reader who half-remembers the pattern loses the cheap reinforcement of reading the comment, seeing the code, and having their understanding confirmed. The goal is neither warning signs everywhere nor minimalism — it's saying the right amount in the right places, tuned for maximum swiftness of understanding. Done well, the presence of a comment itself carries information: something here deserves a beat of your attention, and this sentence is about to make it cheap.

**How versus why.** This tension is about which kind of information a comment carries. If code is genuinely difficult to follow, the first move isn't a comment — it's a rewrite toward code that doesn't need one. Sometimes the difficult form really is right: a recent language feature, a terse chain that's the clearest honest expression, a spot with a genuine pitfall — an exception that won't surface visibly from inside a functional chain, say. Keep those, and comment the mechanics and the trap: that's _how_ work, and it should be light. Everything else — most comments — does _why_ work, the mission comments signed up for: what this line contributes to the task around it, what constraint from elsewhere lands here. And why means why, now: a comment describes what the code does and is for today, never its history or the alternatives it beat — that remembering belongs in an essay.

**Altitude.** This tension is about the height a comment speaks from. Every comment has one — this line, this group, this function, this system — and the failures sit at the extremes. Too low, and it restates the syntax the reader just parsed for themselves. Too high, and a line comment strains to narrate system design: verbose, brittle, and quietly wrong after the next refactor, because nothing nearby will remind anyone to update it. The rule is to match the comment's height to its place. A line comment speaks to its neighborhood — this line, in service of this part of this function. A function comment tells what the function is good for — the need it serves, which is nearly but not quite the story of its callers: describe the need, not the call sites, and the comment survives when the callers change. And when the information is genuinely system-height, don't crush it into a line comment — that's the essay's job.

Comment what's different, not what repeats: in a family of similar functions, comment the first thoroughly — introduce the shared structure there — then comment later members where they diverge, and why. The same for a fancy pattern or unusual platform object used in many places: one full explanation where it lives or first appears, and usage comments that just note it's in play. But mind the radius: an introduction only serves the reader if it's where they'll actually be. A reader lands directly in one function and reads its screenful, its section, its family — not a different chapter three screenfuls up. When a pattern resurfaces beyond that natural viewport, a one-line reintroduction at the new site costs almost nothing; the reader who needed it saves minutes, and the reader who didn't skips a line.

Honest temporary comments are welcome — real work-state lives in the code. The strongest form is the ttd note (things to do): `//ttd march, unify these two paths after the migration`. The token is invented on purpose: it greps clean with no false positives, and types without caps lock, unlike TODO. The month is a timestamp — bare month for this year, month and year like `ttd march2025` for notes surviving from earlier — marking when the issue was raised or last visited. And ttd notes carry special rules: **only the user adds, rewords, or removes them.** A single terse `ttd` can carry weight across sprints and epics — a business concern, a security follow-up, a half-finished design. An agent may move one intact while restructuring, and write around one, but never edits or deletes one — even when deleting the code around it: the structure washes away and the `ttd` comments remain, sitting on the dry riverbed where they landed. If one is in the way, stop and ask; the user will make the edit or give exact instructions permitting it.

Emoji appear sparingly, as landmarks — a marker that makes a section or an essay findable at a glance — never as decoration on every bullet and label. Where a file already uses an emoji pattern, like numbered step markers or a warning mark, maintain it; where emoji appear as accents or fun, leave them be — and don't introduce new decorative ones of your own.

Finally: comments and names are code, and readability is a deliverable. The human path to this quality runs through iteration — finish the logic, pass the tests, then reread as a stranger, renaming and recommenting until understanding comes quickly. You don't need the lengthy process to reach the outcome: write it that way in the first draft. Then still take the stranger's read before calling the work done — the next reader lands on this code cold, and the standing question is: _Could this be any easier to quickly understand?_

## Reserved signals

Two places we invert prevailing style, both for the same reason: when the strict form is rare, its appearance carries information.

- `let` is the default, even for bindings that happen never to change. `const` appears only to draw attention on purpose: this reference should never change, or must never change — factory presets, imports, module-level singletons. This inverts prevailing style, which writes const everywhere and downgrades to let only where reassignment forces it; that habit isn't actually safer or more readable — reassignment bugs are rare, and const leaves the object behind the reference fully mutable.
- `==` and `!=` are the default. `===` and `!==` are reserved for spots where type or coercion genuinely matters — a null-versus-falsy distinction, an api that can hand back mixed types. The strict operator reads as a flag, saying: _be careful right here_. Crockford's Good Parts put == on its bad-parts list, and every linter followed — but the fear, valid in general, is misplaced here: coercion bites when types are uncertain, and this style makes types certain at the boundaries (the guards, below), so inside the walls == compares values whose types are already known. And the deeper practice is designing so the flag is rarely needed: our own functions don't return mixed types to carry extra information. Defaulting to == is table stakes; reserving === for where it matters is the rule; coding so it rarely matters is the goal.

House style — and the linter defaults, prefer-const and eqeqeq — use `const` and `===` everywhere, which spends both signals down to zero.

## Truthiness, text, and absence

- Don't overload falsy with meanings: write `if (o)`, and don't turn undefined, null, and false into three different messages. Where a value crosses a boundary, prefer text-or-blank over flavors of absence — blank means nothing here, and no meanings ride on null versus undefined.
- Zero is a normal number. `if (count)` works, because zero is falsy — but write `if (count > 0)` anyway. The comparison tells the reader that count is a number being measured against a threshold; the bare truthiness test makes them stop and confirm this is the zero-is-falsy trick rather than a presence check. Truthiness shortcuts are for presence, like `if (o)`; numbers get real comparisons.
- Blank is still text. A blank string is falsy, but blank is often a fine value — a successful fetch can return an empty body. So `if (s)` asks two questions at once: is there a string, and is it nonblank. Write code that asks the question it means — a tiny guard of your own, like a hasText(s) that's true only for a string with more than whitespace in it.
- Functions that get something return the thing, or false when not-there is an ordinary answer.

## Exceptions

Where code fails loudly and where it works around trouble quietly is among the most consequential choices in an application's architecture. The rule: loud and brittle for programmer errors; resilient and forgiving for user input, network responses, and third parties. This section and the next are that rule made concrete.

Exceptions are truly exceptional — never an everyday tool, never control flow. Since the 1990s made try/catch fashionable, mainstream code has used exceptions as a second return channel, and paid for it: functions that communicate both by value and by throw, try blocks nested until the logic drifts far from the left margin, and every caller left guessing what might fly out of what. We intentionally left that school, and got simpler code with improved safety — fewer possible paths through the logic means fewer paths nobody reasoned about. Go made the same bet at the language level — errors are values, returned and handled like anything else — and our result objects make it at the level of style.

The architecture is a walled city, with try/catch at exactly two gates:

At the bottom gates, where our code calls down into apis and modules we don't control, a tight try/catch converts exception-speaking code into values. A third-party service throwing or returning 500 is a Tuesday, not a bug — catch it, pin the error into a result object, and let ordinary code decide what happens next: retry, route around, or report failure as a value.

At the top gates — every entrypoint where our code starts running for the framework or platform around it — one broad catch takes anything that escaped, logs it with full context so the team will see it, and fails the request loudly and visibly. An exception reaching this gate means a bug in our code, or input so malformed it suggests tampering; either way the remedy is report-and-fix, not smooth-over-and-retry. A standalone script is the same shape at its smallest: wrap the work in main() and catch once at the bottom — `main().catch(e => { console.error(e); process.exit(1) })`.

Inside the walls, life is calm. Nobody throws for flow and nobody catches: no rethrowing, no try around code that merely feels risky, no dual-channel functions. Happy and sad paths near the user both complete without an exception firing under the hood — wrong password, taken name, rate limited are ordinary answers, returned as result objects like `{success: false, outcome: 'wrongPassword'}`, where the outcome is a short constant the caller can branch on and logs can show. The only deliberate throws inside the city come from guard checks (next section) announcing a bug in the code above — and those fly uncaught to the top gate exactly as intended, because that's what they are: our mistake, to be fixed in code, not negotiated with at runtime.

What this buys: every function inside communicates one way, by return value, so control flow is visible on the page — reading a function, you can see every exit. Code stays near the left margin instead of nesting through pyramids of try blocks. And safety improves twice over — the top gate means nothing escapes unreported, and the simpler flow means fewer control-flow bugs to escape in the first place.

## Guards and trust

Important functions check their inputs at the top. A line of checks there is a load test at a gate: any imaginable correct use passes; a call that trips one has revealed a bug in the code above, caught at the boundary instead of corrupting quietly below. The check throws — that's the programmer-error loud path from the section above — and carries the named variables involved, so the mistake arrives in the log with its context. Checks encode intent, narrower than mechanics: a function might run fine on a blank string, but if callers should never pass blank, a check at the top catches the upstream bug the moment it happens, right where it happened. These are preconditions — design by contract's oldest tool — enforced at runtime, where they hold against the actual data of this call, not just the declared shape of the code. In a young project these begin as plain lines — `if (typeof s != 'string' || !s.trim()) throw new Error('expected text')` — and as the same checks repeat, they earn names and become the project's small guard family, keeping the prefix promises from the names section: has* and is* to ask, check* to enforce. The family is grown out of repetition, not installed on day one.

Defense belongs at trust boundaries, not everywhere. Checks, optional chaining, default arguments, and fallbacks are load-bearing where untrusted data enters — http bodies, decrypted payloads, provider responses. Inside trusted code they're noise: they hide where the real defense is and erode the very contracts the checks establish.

No cargo culting, no magical thinking. Cargo culting is copying the form of a thing without its function — named for the post-war islanders who built bamboo control towers hoping the supply planes would return. In code it's the header copied from a tutorial, the defensive check nobody can explain, the config flag that was in the example — parts kept because they were seen near working code, not because anyone traced what they do. Every part here has a reason to be there, and when we're unsure whether a part is necessary, we find out — read the documentation, read the code beneath, run a quick experiment in the environment where the behavior actually matters — rather than leaving it in just in case. This is what prevents the growing garbage patch of parts nobody dares remove.

We never polyfill and never probe for whether the platform supports a feature — we target current browsers and current node, and code to them. One narrow exception: a permanent test proving, once, that a critical piece behaves exactly to standard — reserved for pieces that would fail silently. Here's the case that earned the rule. Code that encodes binary data as text needs an alphabet: a string literal like '0123456789ABC...xyz' listing which character stands for which value, typed into the code or pasted from a reference. Swap two characters in that string and nothing visibly breaks — encoding still runs, decoding still round-trips, the app's own tests still pass — but every value stored is now written in a private dialect, silently incompatible with the standard and with any other implementation, and the damage compounds with every row written until someone notices, possibly years later, from the outside. So that alphabet gets a permanent test with fixed inputs and known correct outputs taken from the published standard: encode exactly these bytes, expect exactly this text. That's the shape of the exception — pasted constants, encoding tables, padding and byte-order details, anywhere a platform or a paste could be wrong without anything failing loudly — a one-time check against known answers. Everything else, trust the platform.

Logging: temporary logs while building something are fine — part of watching a pattern come alive — and removed once it's confirmed working. Permanent logs are different: not print statements but records of system behavior, placed where things could go wrong or where a rare, important pathway runs, so that when something does go wrong, there's a chance of seeing it later.

## Factoring

Use the minimum of the framework, and keep application logic out of it. The novice pattern goes the other way: logic is born inside a component, and when a second component needs it, the answer gets searched for in the framework's documentation — some more complex, more framework-specific sharing pattern — instead of factoring the logic down into a plain module both components import. Follow that road to the end and the application is a christmas tree: framework as the trunk, application logic hung on every branch, inseparable from it. When the framework's next major version lands, or the business picks a different framework, the whole tree goes out to the curb. Factored our way — thin framework files calling down into plain JavaScript modules — the logic survives the transition, and only the thin layer is rewritten.

- Factor down, and all the way: pages, components, and endpoints stay thin; logic settles into library files where it's isomorphic — the same validateEmail correcting input on the page and guarding the database write.
- The hard core of a difficult thing is a pure function, wrapped by a thin function that does the platform and storage work. Pure cores are trivially testable.
- Plain objects over classes: build `let o = {}`, fill it as work proceeds, and write families of functions that operate on the shared shape. Crockford drew the same line — the object literal is among the language's genuinely good parts; the class ceremony is borrowed clothing. Related: when a function takes several parameters, pass one object with named properties — self-labeling at the call site, immune to ordering mistakes, easy to extend.
- The constellation pattern: an important feature is a vertical column across the stack — a table schema, the functions on it, an endpoint, a panel component — and we keep the column together in fewer, longer files, with region comments as chapter headings. The tiny-functions orthodoxy pushes the opposite way — functions of a few lines, a file per small class — and followed faithfully it shatters a feature into confetti: twenty files that each say almost nothing, the logic living in the seams where no one can read it. Carmack argued our side from games: keep code long and in execution order, because reading what actually happens beats hopping between fragments. We keep the feature whole. Where the framework mandates structure, like a routes directory or one component per file, comply — and keep those mandated files thin, calling down into the long plain files whose structure we control. The payoff is diff locality: code that changes together lives together, so a feature's branch touches one or two files, not twenty.
- Refactor together, and completely: when usage shifts or a better pattern emerges, rename and move immediately so the whole codebase reflects present understanding. Half-refactored artifacts and vestigial layers make appearances impossible to trust. And when a cleanup fixes an anti-pattern in one spot, sweep the neighbors for the same pattern before calling it done.
