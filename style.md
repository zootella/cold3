# JavaScript Style Guide

This is the style guide for coding and editing JavaScript here — the user telling you how to write it. It governs your choices about coding, commenting, documenting, and refactoring. Your default modern style is already good; this document is the delta: where we deliberately differ from prevailing practice, and why. Where it's silent, good current practice applies.

By JavaScript we mean code and comments as one integrated system. The runtime reads only the code; every other reader reads the two as a single text. That's always been true for humans, and now it's true for agents: drop a capable agent into a repo with the comments stripped out, and its results suffer, because the comments carry invaluable context and meaning. The aim everywhere: code that runs reliably, is factored efficiently, and reads lyrically, for humans and agents alike.

Two ideas underneath everything here. First, **simplicity**: we write simple, modern JavaScript — the current language, used plainly. Not bleeding-edge syntax that shows off, and not the naive patterns of a first tutorial either: simplicity within what's modern and current. Python states simplicity as a core value — simple is better than complex — and JavaScript should hold the same. Reach for the plain pattern over the fancy one, and when code grows complicated, break it into smaller, plainer parts rather than adding cleverness. Second, **attention**: a reader's attention is what every choice here spends or serves. That's not a rule about writing less — comments enjoy a huge budget in this style; by weight they often rival the code! It's a rule about balance: say the right amount in the right places, so understanding arrives as quickly and accurately as possible.

This document positions itself on the map of styles you already know — your own house style, the Airbnb style guide taught in bootcamps, Prettier and the linter defaults, Clean Code and The Good Parts. We take the best ideas from these schools, but where we oppose one, the opposition is deliberate, and argued here with reasons. Knowing what the mainstream does, and choosing with intent, is part of understanding this style.

## Layout

- Indent with tabs; editors show them two wide.
- Never wrap lines to a column width; assume a graphical editor with soft wrap on. Artificial wrapping turns a one-word change into a multi-line diff, and wastes our effort counting columns and choosing break points.
- Do align neighboring lines vertically: when adjacent lines share structure, pad with spaces so like parts sit in the same columns. The eye diffs aligned lines instantly — seeing the differences is understanding them. Together, this rule and the one above ban formatters: a tool that owns the whitespace erases information we deliberately write into it.
- Use single quotes for strings, backticks when interpolating, and double quotes inside comments and prose.
- Omit semicolons except where one is necessary for execution — the main case is two statements sharing a line because they're one thought: `x = 0; y = 0//reset the cursor to the origin`. This is the same reservation that governs `const` and `===` below: _absent_ by default, _present_ with meaning.
- Write a short `if` on one line without braces: `if (!row) return false` — or with braces holding two short statements: `if (found) { count++; last = x }`. A guard and its action on one line is a single readable sentence.
- Prefer `if` statements over ternaries. Let a ternary survive only when it's flat and simple, picking between two values — never nested, never carrying logic.
- End every line of a multi-line literal with a trailing comma, so reordering or adding a line never touches its neighbor.
- Give blocks spaces inside their braces `{ return false }`; give object literals none `{key: 'value'}`.
- Write comments in conversational lowercase, for the same reason syntax highlighting paints them dim gray: to let the code, and its capitalization, stand out. The one exception is the essay, described with the other comment forms below, which is written in a `/* */` block in normal English prose.

## Names

- Name things with complete, speakable words. **No abbreviations** — _configuration_ and _c_ are acceptable, _cfg_ is not. Treat an abbreviation that's a proper name out in the world, like HTML or URL, as a word: _someUrl_, _someHtml_, _someId_.
- Name functions _verbNoun_, like _validateEmail_ and _parseHeader_; name getters as bare nouns.
- Treat a verb prefix as a promise: if `is`* functions answer yes-or-no, every `is`* function answers yes-or-no. A reader who has learned one member of a family has learned them all, so never break a prefix's contract.
- When a family of functions shares a domain, lead their names with the shared noun so they sort and read together: _credentialOauthSet_, _credentialOauthGet_, _credentialOauthRemove_.
- Mark a private helper — a function serving a nearby one — with a leading underscore.
- Use single-character names in exactly two places. First, conventional carriers a codebase establishes for itself — picture a project where every validation function returns a result object the code always calls _v_; after you've met _v_ twice, it never needs introducing again. Second, short-lived locals whose whole life fits on a screen — introduce these with a comment; the comment is the real description, and a long name would be a worse, endlessly repeated version of it. Prevailing style would lengthen the name until it describes itself; ours goes the other way: name it briefly, describe it richly, once, in the comment where it's born. A name is repeated at every use; a description shouldn't be.
- Follow the environment's case: `kebab-case` in CSS, `snake_case` in SQL (our_tables have column_names), `camelCase` in JS. **No ALL_CAPITALS** — those read as a shout and are clumsy to type. Where you'd normally use SCREAMING_SNAKE on a constant, write `const camelCase` at the top of the file instead.

The rules in this guide have exceptions — you'll meet them in existing code, and sometimes the right move is to write one. Is _maximum_ really required instead of just _max_? No: _max_ and _min_ are words programmers speak, and the rule against abbreviations was never against words. Decide cases like this by returning to what the rule protects — quick, accurate understanding — and choosing what serves it. Almost always that's the rule as written; occasionally it's a small, obvious exception, taken without ceremony; and when a case genuinely pulls both ways and the stakes are real, ask the user. Never follow a rule into code you know is worse, and never queue up permission requests for calls you can make — both mistake what this document is: values first, with rules as their everyday form.

## Comments

The signature of this style is comment density: get ready to comment far more than you normally would, and differently. This is a considered rejection of an orthodoxy you've absorbed. _Clean Code_ taught that a comment is a failure to express yourself in code, and a generation of review culture inherited the reflex — comments as apologies, deleted on sight, replaced by ever-longer names and ever-smaller functions. We hold with the older and better school — Knuth called it _literate programming_, a program written for people and only incidentally for the machine: prose carries intent, context, and caution that code cannot, and refusing that channel throws away half the bandwidth of the file. Write toward this bar: comments so complete that given only a function's name, parameters, and comments — body stripped away — a capable engineer or model could write a functional equivalent.

Put the comment at the end of the line it's about, no space around the slashes:

	let a = []//the array we'll fill with results, sort, and return

Write it lowercase, with no closing period. Separate two clauses with a semicolon:

	//the address is gmail; apply the extra normalization rules

Let several sentences form a short paragraph — capitals and periods between sentences, but the same lowercase open and bare end:

	//the first note. And a second, a clarification. Perhaps a third situation to describe, as well

Comments come in four sizes, small to large: on the end of a line; introducing a group of lines; above a function; and the essay.

The first two live inside function bodies. Put line comments to the right of code, not above, so comments stay plentiful without lengthening the file. Split a longer function into conceptual groups with blank lines; the reader sees two groups and knows the function works in two parts, then scans deeper to learn what each part is. Put a single-line comment above a group when it speaks at the group's altitude — what these lines accomplish together. Put a comment about one line on that line.

Above a function, write the same single-line style one step zoomed out: what this is for, plus anything clever, unique, or tricky. Javadoc-style blocks are the anti-pattern — formulaic and pedantic, covering the trivial and the tricky at equal weight until the important part is lost in the average, sitting atop code the reader still has to figure out. Comment instead like the engineer who knows this area talking to a capable newcomer, saying: _here's what you actually need to know_.

Fourth, the essay. Write one when a larger system needs explaining at a larger level — an overall design, a novel mechanism running full stack, choices and their reasons. The comment grows into a `/* */` block and switches to normal English prose, capitals and all. Write it like a little document, but keep it with the code, at the best place it can sit, and keep it a half screenful, not pages. The essay speaks from ten thousand feet, and it's the one place comments are allowed to remember — the history, how we got here, the alternatives weighed and set aside. Essays span larger time and space; keep every smaller comment tight to here and now.

Never restate the line. Positioned at the line, a comment never has to explain where it applies — the reader is already there — so spend all its words on what that line is doing in service of the surrounding task. Writing good comments means balancing three tensions, deliberately:

**Comment, or let code stand alone.** The test is whether the comment speeds understanding: would a reader move through this stretch faster, easier, and more accurately with it than without it? Trivial code commented is clutter — it slows the eye and dilutes the comments that matter. But an unfamiliar pattern uncommented is worse: every future reader stops to reverse-engineer what one sentence would have handed them, and a reader who half-remembers the pattern loses the cheap reinforcement of reading the comment, seeing the code, and having their understanding confirmed. Aim for neither warning signs everywhere nor minimalism — say the right amount in the right places, tuned for maximum swiftness of understanding. Done well, the presence of a comment itself carries information: something here deserves a moment of your attention, and these words exist to make that moment short.

**How versus why.** This tension is about which kind of information a comment carries. If code is genuinely difficult to follow, don't reach for a comment first — rewrite toward code that doesn't need one! Sometimes the difficult form really is right: a recent language feature, a terse chain that's the clearest honest expression, a spot with a genuine pitfall — an exception that won't surface visibly from inside a functional chain. Keep those, and comment the mechanics and the trap: that's _how_ work, and keep it light. Spend most comments on _why_ work, the mission the comments signed up for: what this line contributes to the task around it, what constraint from elsewhere lands here. And the _why_ is always today's _why_: describe what the code does and is for right now, never its history and seldom the alternatives it beat — that remembering belongs in an essay.

**Altitude.** This tension is about the height a comment speaks from. Every comment has one — this line, this group, this function, this system — and the failures sit at the extremes. Too low, and it restates the syntax the reader just parsed for themselves. Too high, and a line comment strains to narrate system design: verbose, brittle, and quietly wrong after the next refactor, because nothing nearby will remind anyone to update it. Match the comment's height to its place. Speak a line comment to its neighborhood — this line, in service of this part of this function. Have a function comment tell what the function is good for — the need it serves, which is nearly but not quite the story of its callers: describe the need, not the call sites, and the comment survives when the callers change. And when the information is genuinely system-height, don't crush it into a line comment — that's the essay's job.

Comment what's different, not what repeats: in a family of similar functions, comment the first thoroughly — introduce the shared structure there — then comment later members where they diverge, and why. Do the same for a fancy pattern or unusual platform object used in many places: one full explanation where it lives or first appears, and usage comments that just note it's in play. But mind the radius: an introduction only serves the reader if it's where they'll actually be. A reader lands directly in one function and reads its screenful, its section, its family — not a different chapter three screenfuls up. When a pattern resurfaces beyond that natural viewport, add a one-line reintroduction at the new site — it costs almost nothing; the reader who needed it saves minutes, and the reader who didn't skips a line.

Welcome honest temporary comments — real work-state lives in the code. The strongest form is the `ttd` note (things to do): `//ttd march, unify these two paths after the migration`. The token is invented on purpose: it greps clean with no false positives, and types without caps lock, unlike TODO. The month is a timestamp — bare month for this year, month and year like `ttd march2025` for notes surviving from earlier — marking when the issue was raised or last visited. And ttd notes carry special rules: **only the user adds, rewords, or removes them.** A single terse `ttd` can carry weight across sprints and epics — a business concern, a security follow-up, a half-finished design. Move one intact while restructuring, write around one, but never edit or delete one — even when deleting the code around it: the structure washes away and the `ttd` comments remain, sitting on the dry riverbed where they landed. If one is in the way, stop and ask; the user will make the edit or give exact instructions permitting it.

Existing code may use emoji sparingly, as landmarks — a marker that makes a section or an essay findable at a glance. Where a file already uses an emoji pattern, like bulleted step markers or warning marks, maintain it; where emoji appear as accents or fun, leave them be — and don't introduce new decorative ones of your own.

Finally: comments and names are code, and readability is a deliverable. The human path to this quality runs through iteration — finish the logic, pass the tests, then reread as a stranger, renaming and recommenting until understanding comes quickly. You don't need the lengthy process to reach the outcome: write it that way in the first draft. Then still take the stranger's read before calling the work done — the next reader lands on this code cold, and the standing question is: _Could this be any easier to quickly and accurately understand?_

## Reserved signals

Two places we invert prevailing style, both for the same reason: when the strict form is rare, its appearance carries information.

- Default to `let`, even for bindings that happen never to change. Reserve `const` to draw attention on purpose: this reference should never change, or must never change — factory presets, imports, module-level singletons. This inverts prevailing style, which writes `const` everywhere and downgrades to `let` only where reassignment forces it; that habit isn't actually safer or more readable — reassignment bugs are rare, and `const` leaves the object behind the reference fully mutable.
- **Never return mixed types to carry extra information.** With returns in one shape, and the guards below checking inputs, types are already certain wherever values meet — so default to `==` and `!=`; coercion has nothing to bite. Reserve `===` and `!==` for the rare genuine exception — a null-versus-falsy distinction, a mixed-type api you don't control — where the strict operator's rarity reads as a flag: _be careful right here_. Crockford's _Good Parts_ put `==` on its bad-parts list and every linter followed; that fear is valid where types are uncertain. Ours aren't.

Prevailing style uses `const` and `===` everywhere, which spends both signals down to zero.

## Truthiness, text, and absence

- Don't overload falsy with meanings: write `if (o)`, and don't turn _undefined_, _null_, and _false_ into different messages. Where a value crosses a boundary, prefer text-or-blank over flavors of absence: a blank string simply means there's nothing, and no extra meanings ride on null versus undefined. Imagine the caller is going to test your return for falsiness, and that is all.
- Zero is a normal number. `if (count)` works, because zero is falsy — but write `if (count > 0)` anyway. The comparison tells the reader that count is a number being measured against a threshold; the bare truthiness test makes them stop and confirm this is the zero-is-falsy trick rather than a presence check. Keep truthiness shortcuts for presence, like `if (o)`; give numbers real comparisons.
- Blank is still text. A blank string is falsy, but blank is often a fine value — a successful fetch can return an empty body. So `if (s)` asks two questions at once: is there a string, and is it nonblank. Write code that asks the one question it means — a tiny guard of your own, like a hasText(s) that's true only for a string with more than whitespace in it. This is about asking clearly, not answering cleverly: never flip it around and make false, blank, and text three different messages from one return — that's exactly the falsy overloading banned above.
- Have get-style functions return the thing, or false when not-there is an ordinary answer. This isn't the mixed-type return banned above: false carries only absence, and the caller tests falsiness — that is all.

## Exceptions

Where code fails loudly and where it works around trouble quietly is among the most consequential choices in an application's architecture. The rule: loud and brittle for programmer errors; resilient and forgiving for user input, network responses, and third parties. This section and the next are that rule made concrete.

Exceptions are truly exceptional — never an everyday tool, never control flow. Since the 1990s made _try/catch_ available, and thus fashionable, mainstream code has used exceptions as a second return channel, and paid for it: functions that communicate both by value and by throw, try blocks nested until the logic drifts far from the left margin, and every caller left guessing what might fly out of what. We intentionally left that school, and got simpler code with improved safety — fewer possible paths through the logic means fewer paths nobody reasoned about. _Go_ made the same bet at the language level — errors are values, returned and handled like anything else — and our result objects make it at the level of style.

Build the architecture as a walled city, with try/catch at exactly two gates:

At the bottom gates, where our code calls down into apis and modules we don't control, wrap the call in a tight try/catch that converts exception-speaking code into values. A third-party service throwing or returning 500 is a Tuesday, not a bug — catch it, pin the error into a result object, and let ordinary code decide what happens next: retry, route around, or report failure as a value.

At the top gates — every entrypoint where our code starts running for the framework or platform around it — write one broad catch: log anything that escaped, with full context so the team will see it, and fail the request loudly and visibly. An exception reaching this gate means a bug in our code, or input so malformed it suggests tampering; either way the remedy is report-and-fix, not smooth-over-and-retry. Wrap a standalone script the same way at its smallest: put the work in main() and catch once at the bottom — `main().catch(e => { console.error(e); process.exit(1) })`.

Inside the city walls of the code we write and control, life is calm. Never throw for flow and never catch: no rethrowing, no try around code that merely "feels risky", no dual-channel functions that answer both by return and by throw. Let happy and sad paths near the user complete without an exception firing under the hood — wrong password, taken name, rate limited are ordinary answers; return them as result objects like `{success: false, outcome: 'WrongPassword.'}`, where the outcome is a short constant the caller can branch on and logs can show. The only deliberate throws inside the city come from guard checks (next section) announcing a bug in the code above — let those fly uncaught to the top gate, exactly as intended, because that's what they are: our mistake, to be fixed in code, not negotiated with at runtime.

What this buys: every function inside communicates one way, by return value, so control flow is visible on the page — reading a function, you can see every exit. Code stays near the left margin instead of nesting through pyramids of try blocks. And safety improves twice over — the top gate means nothing escapes unreported, and the simpler flow means fewer control-flow bugs to escape in the first place.

## Guards and trust

Check inputs at the top of important functions. A line of checks there is a gate that every call must pass through: any imaginable correct use passes untouched; a call that trips one has revealed a bug in the code above, caught at the boundary instead of corrupting quietly below. Make the check throw — that's the programmer-error loud path from the section above — and have it carry the named variables involved, so the mistake arrives in the log with its context. Write checks that encode _intent_, narrower than _mechanics_: a function might run fine on a blank string, but if callers should never pass blank, a check at the top catches the upstream bug the moment it happens, right where it happened. These are preconditions, enforced at runtime, where they hold against the actual data of this call, not just the declared shape of the code. In a young project, write them as plain lines — `if (typeof s != 'string' || !s.trim()) throw new Error('expected text: ' + s)` — and as the same checks repeat, promote them into the project's small guard family, keeping the prefix promises from the names section: has* and is* to ask, check* to enforce. Grow the family out of repetition; don't install it on day one.

Put defense at trust boundaries, not everywhere. Checks, optional chaining, default arguments, and fallbacks are load-bearing where untrusted data enters — HTTP bodies, decrypted payloads, provider responses. Keep them out of trusted code, where they're noise: they hide where the real defense is and erode the very contracts the checks establish.

Never keep a part just because it was seen near working code. The header copied from a tutorial, the defensive check nobody can explain, the config flag that was in the example — these are parts wearing the form of working code without anyone having traced their function. Give every part a reason to be there, and when you're unsure whether a part is necessary, find out — read the documentation, read the code beneath, run a quick experiment in the environment where the behavior actually matters — rather than leaving it in just in case. This is what prevents the growing garbage patch of parts nobody dares remove.

Never polyfill, and never write code that checks whether the platform supports a feature — our projects target current browsers and recent Node, so the features are there; use them directly. The one exception is behavior that would fail silently: most mistakes announce themselves by throwing or visibly breaking, but a few — an encoding detail, a pasted constant — can be wrong and still run, quietly corrupting the data they touch. Pin behavior like that with a permanent test against known correct answers: feed in fixed inputs, expect the exact outputs the standard publishes. Trust the platform for everything else.

Log freely while building something — that's part of watching a pattern come alive — and remove those logs once it's confirmed working. Treat permanent logs differently: they're not print statements but records of system behavior. Place them where things could go wrong, or where a rare, important pathway runs, so that when something does go wrong, there's a chance of seeing it later.

## Factoring

Use the minimum of the framework, and keep application logic away from it. The novice pattern goes the other way: logic is born inside a component, and when a second component needs it, the answer gets searched for in the framework's documentation — some more complex, more framework-specific sharing pattern — instead of factoring the logic down into a plain module both components import. Follow that road to the end and the application is like a christmas tree: framework as the trunk and branches, application logic hung on every branch, inseparable from it. When the framework's next major version lands, or the business picks a different framework, the whole project goes out to the curb. Factored our way — thin framework files calling down into plain JavaScript modules — the logic survives the transition, and only the thin layer is rewritten.

- Factor down, and all the way: keep pages, components, and endpoints thin; settle logic into library files that are isomorphic — the same code running on page and server — so one _validateEmail_ function corrects input on the page and guards the database write exactly the same way.
- Make the hard core of a difficult thing a pure function, wrapped by a thin function that does the platform and storage work. Pure cores are trivially testable.
- Choose plain objects over classes: build `let o = {}`, fill it in the lines that follow, and write families of functions that operate on the shared shape. Related: when a function takes several parameters, pass one object with named properties — self-labeling at the call site, immune to ordering mistakes, easy to extend.
- Prefer fewer, longer files, with region comments as chapter headings. An important feature is a vertical column through the stack — a table schema, the functions on it, an endpoint, a panel component — and that column belongs together in one long file, not scattered across twenty tiny files where the logic lives in the seams between them. Where the framework requires its own file structure, like a routes directory or one component per file, comply — and keep those required files thin, calling down into the long files we control. The payoff: code that changes together lives together, so a feature's diff touches one or two files, not twenty.
- Refactor together, and completely: when usage shifts or a better pattern emerges, confer scope with the user, then rename and move immediately so the whole codebase reflects present understanding. Half-refactored artifacts and vestigial layers make appearances impossible to trust. And when a cleanup fixes an anti-pattern in one spot, sweep the neighbors for the same pattern before calling it done.
